THREADSPACE
Complete Technical Specification
Graph-native identity discovery · Stack · Algorithms · Deployment
1. Vision and Core Idea
ThreadSpace is a graph-native identity discovery platform. Every person is a structured node in a knowledge graph. Every preference, trait, fandom, and personality marker is a connected sub-node. Every relationship between those markers is a typed, weighted edge. The entire product is one canonical graph, and the website is a controlled way of walking that graph.
The practical result is something that feels like a physically explorable map rather than a social network feed. A user does not browse profile cards. They step through a living ontology where each click narrows the space, increases similarity, or reveals a more specific thread. Two roles exist: the Hider, who creates one permanent identity card and populates it with traits, and the Finder, who explores the graph without necessarily being logged in. This asymmetry is deliberate. It preserves the game-like, exploratory nature of the product while preventing spam and identity abuse.
Every feature — friend discovery, hot-cold matching, niche group detection, challenge links — is a constrained traversal of the same graph. There are no parallel data models glued together. There is one source of truth. If that rule breaks, the product loses its core idea.

2. Final Finalized Stack
The chosen stack is: Next.js 15 App Router with React 19 and TypeScript for the full application framework; Tailwind CSS v4 with shadcn/ui for the design system and component primitives; Motion (formerly Framer Motion) for all animations; @xyflow/react (React Flow) for the graph canvas UI; Neon serverless Postgres for the database (replacing Supabase Postgres because Neon does not pause on inactivity and has a more generous free tier); Better Auth for authentication (replacing Supabase Auth because Better Auth is fully open-source, TypeScript-native, self-hostable, supports passkeys, OAuth, magic links, and session management without vendor lock-in); Drizzle ORM for typed schema and migrations; and Vercel Hobby for frontend deployment.
The reason this stack holds together is clean layer separation. Next.js owns rendering and the server-side mutation path via Server Actions and Route Handlers. React owns component composition. Tailwind and shadcn own the visual system. Motion owns transitions. React Flow owns the graph canvas. Neon owns durable relational storage. Drizzle owns the typed schema contract. Better Auth owns session and identity. The graph traversal and scoring logic lives as SQL inside Neon — not as a separate service. This keeps the architecture stateless on the frontend, indexed in the database, and graph-aware in how it fetches only the neighborhood it needs.
For free deployment, Vercel Hobby gives a permanent free tier with edge deployment globally and a vercel.app domain. Neon's free tier gives 0.5 GB of storage, no connection limits via the serverless driver, and critically no inactivity pausing. This combination is zero-cost for at least one full year of MVP operation at typical early-stage traffic. No credit card is required for either. When traffic forces an upgrade, the architecture is already stateless and the database schema is already graph-shaped, so migration is straightforward.

3. Database Schema and Graph Model
3.1 Core Philosophy
PostgreSQL represents the graph through node and edge tables rather than a specialized graph database. This gives durability, mature indexing, strong consistency, and free hosting without early lock-in. The schema is graph-shaped from day one so the storage engine can evolve — to Neo4j or a hybrid — without rewriting the product logic.
3.2 Table Definitions
USERS TABLE
Stores the authentication identity. Fields: id (uuid primary key), email (unique, not null), created_at (timestamptz default now), updated_at (timestamptz). This table is the root of all ownership. One user owns exactly one identity card.
SESSIONS TABLE
Managed by Better Auth automatically. Fields: id (uuid), user_id (uuid foreign key to users), token (text, hashed), expires_at (timestamptz), created_at. Better Auth handles all session lifecycle including refresh and revocation. Row-level security ensures users can only read their own session rows.
IDENTITY_CARDS TABLE
The Hider's primary artifact. Fields: id (uuid primary key), user_id (uuid unique foreign key to users — one card per user, enforced at schema level), display_name (text), avatar_url (text nullable), visibility (enum: public, private, friends_only, default public), created_at, updated_at, version (integer default 1). The unique constraint on user_id is the schema-level enforcement of the one-card-per-person rule. Version is incremented on every mutation for optimistic concurrency.
NODES TABLE
The universal node store. Every person, show, character, trait, concept, community, or derived category is a row here. Fields: id (uuid primary key), type (enum: person, show, character, trait, concept, community, derived), slug (text unique — normalized lowercase kebab-case identifier), display_name (text), metadata (jsonb — type-specific extra data), visibility (enum: public, private), created_at, updated_at. The slug is the canonical identifier used in URLs and search indexes. Indexes: btree on type, btree on slug, gin on metadata for jsonb queries.
EDGES TABLE
The universal edge store. Every relationship between two nodes is a row here. Fields: id (uuid primary key), source_id (uuid foreign key to nodes), target_id (uuid foreign key to nodes), edge_type (text — e.g. likes, has_trait, similar_to, belongs_to, inferred_affinity), weight (float4 default 1.0), confidence (float4 default 1.0 — 1.0 for explicit user input, 0.3 to 0.7 for system-inferred), source_of_truth (enum: user_explicit, system_inferred, admin), created_at, updated_at. Indexes: btree on (source_id, edge_type), btree on (target_id, edge_type), btree on confidence, composite btree on (source_id, target_id) for dedup.
CARD_NODES TABLE
Junction table linking an identity card to the nodes it claims. Fields: id (uuid), card_id (uuid foreign key to identity_cards), node_id (uuid foreign key to nodes), visibility (enum: public, friends_only, hidden), added_at (timestamptz). This allows per-trait visibility control. A Hider can make their love of a niche show public but keep their MBTI type friends-only. Index: btree on (card_id, node_id).
TFIDF_VECTORS TABLE
Precomputed TF-IDF sparse vector store. Fields: card_id (uuid primary key), vector (float4[] — sparse array indexed by node dimension), vector_norm (float4 — precomputed magnitude for cosine similarity), computed_at (timestamptz). Updated asynchronously by background job on card mutation. This table is the mathematical heart of the similarity engine.
NODE_COOCCURRENCE TABLE
Aggregate co-occurrence counts used by the ontology inference job. Fields: node_a (uuid), node_b (uuid), co_count (integer), total_cards_seen (integer), updated_at. Primary key on (node_a, node_b). The job reads from card_nodes and writes aggregate counts here. When co_count / total_cards_seen exceeds the promotion threshold, a soft inferred edge is created in the edges table.
AUDIT_EVENTS TABLE
Append-only event log for all graph mutations. Fields: id (uuid), user_id (uuid nullable), event_type (text), payload (jsonb), created_at. Used for debugging, rate limiting, and future behavioral signal processing. Never deleted.
SEARCH_TERMS TABLE
Denormalized search index for fast autocomplete. Fields: node_id (uuid), term (text), term_tsvector (tsvector), weight (float4). Populated asynchronously from node mutations. The tsvector column uses GIN index for full-text search. The trigram extension (pg_trgm) is enabled on the term column for fuzzy matching.

4. Authentication and Session Architecture
4.1 Better Auth Setup
Better Auth is installed as a Next.js plugin. The auth configuration lives in lib/auth.ts and is instantiated once at module load. It connects directly to Neon via the Drizzle adapter, meaning Better Auth manages its own tables (users, sessions, accounts, verifications) inside the same Postgres database. No separate auth database is needed.
The auth configuration enables the following providers: email and password with bcrypt hashing (cost factor 12); magic link via email using Resend's free tier (100 emails per day, sufficient for MVP); GitHub OAuth for social login; and passkey support via WebAuthn for 2026-standard passwordless authentication. Each provider is toggled in the Better Auth config object, requiring only the relevant environment variables.
4.2 Session Handling
Better Auth issues httpOnly secure cookies for session tokens. The session token is a random 32-byte hex string stored hashed in the sessions table. On each request, the Next.js middleware reads the cookie, verifies the hash against the database, and attaches the session to the request context. The middleware runs at the edge, meaning session verification adds negligible latency. Sessions expire after 30 days by default, with a sliding expiry that resets on each active request.
Anonymous Finder access requires no session. The Next.js middleware distinguishes between authenticated requests (session cookie present and valid) and anonymous requests. Anonymous requests can read public nodes and traverse the graph but cannot mutate any data. Server Actions check session validity as their first operation before any database write. There is no route that allows unauthenticated writes.
4.3 First Login and Card Creation
When a user authenticates for the first time, Better Auth creates a row in users. A Next.js Server Action triggered immediately after first auth checks whether an identity_cards row exists for that user_id. If none exists, it creates a blank card with visibility set to private. The user is redirected to the Identity Card Editor, where the onboarding flow begins. The card stays private until the user explicitly publishes it. This prevents half-finished cards from appearing in the graph.
Re-login is seamless. Because all identity data is stored in Neon and the session is restored from the cookie, the user lands exactly where they left off — their current card state, recent traversal history stored in session state, and any saved searches. No client-side local storage is used; everything is server-authoritative.
4.4 Row-Level Security
Neon supports PostgreSQL row-level security policies. Three policies are defined: users can select, update, and delete only rows in identity_cards, card_nodes, and tfidf_vectors where user_id matches their session user_id; all users (including anonymous) can select rows in nodes and edges where visibility is public; only service-role connections (used by background jobs) can insert into node_cooccurrence, tfidf_vectors, and edges with source_of_truth of system_inferred. These policies are the last line of defense and enforce correctness even if application-level auth logic has a bug.

5. Identity Card — Creation, Storage, and Updates
5.1 Onboarding Flow
The Identity Card Editor is a multi-step form built with shadcn/ui components and Tailwind v4. It is not a graph canvas — it is a structured form that maps user input directly to node and edge creation. The form is divided into sections: Display Name and Avatar (stored on the identity_cards row); Favorite Shows and Media (creates or connects to existing show nodes via slug lookup); Favorite Characters (creates character nodes linked to their show nodes); Personality Traits and Descriptors (connects to existing trait nodes or creates new ones with admin review flag); Questionnaire (short-answer prompts whose responses are parsed into node connections); and Visibility Settings (per-section toggle for public, friends-only, or hidden).
Each form step executes a Server Action on submit. The Server Action does three things atomically in a Postgres transaction: upsert the relevant node rows (using slug as the dedup key — if a show node for friends-sitcom-friends already exists, it is reused, not duplicated); insert rows into card_nodes linking the card to those nodes; and insert rows into edges linking node-to-node relationships (character belongs-to show, trait has-confidence 1.0, etc.). The transaction either fully succeeds or fully rolls back. The card version number is incremented on each successful transaction.
5.2 Node Deduplication
Slug normalization is the dedup mechanism. Before any node insert, the Server Action runs slug = text.toLowerCase().replace(/[^a-z0-9]+/g, '-').trim(). It then attempts an INSERT INTO nodes ... ON CONFLICT (slug) DO UPDATE SET updated_at = now() RETURNING id. This is a single SQL statement that either creates or touches an existing node and returns its id in one round-trip. The returned id is used for the card_nodes and edges inserts that follow.
5.3 Card Updates
Card updates follow the same transactional pattern as creation. A Server Action receives the mutation (add trait, remove show, change visibility), opens a transaction, performs the relevant node upsert and card_nodes insert or delete, increments the version number, and commits. The version number enables optimistic concurrency: if two browser tabs attempt concurrent mutations, the second one's version check fails and it receives a conflict error, prompting a refresh. This prevents silent data corruption without requiring pessimistic locks.
Avatar upload uses Cloudflare R2's free tier (10 GB storage, 1 million read requests per month, free egress — genuinely free, not trial). The Server Action generates a presigned upload URL via the R2 S3-compatible API, returns it to the client, and the client uploads directly to R2 from the browser. The avatar_url field on identity_cards is updated only after the upload succeeds, verified by a separate Server Action that receives the upload confirmation.

6. Graph Traversal and Hot-Cold Scoring
6.1 Traversal Architecture
All graph traversal happens in a single PostgreSQL recursive CTE query executed against Neon. There is no graph database, no Neo4j, no separate traversal service at MVP. The recursive CTE walks the edges table outward from a set of seed node IDs, collecting candidate card IDs at each hop, up to a maximum depth of three hops. This query is the engine of the entire hot-cold system.
The traversal CTE has three parts. The base case selects all card_node rows where the node_id matches any seed node. The recursive case joins edges to find neighboring nodes, then joins card_nodes to find cards that contain those neighboring nodes, accumulating hop count and path metadata. The final select aggregates per candidate card, computing the composite score.
6.2 Composite Score Formula
The score for each candidate card is computed as the sum over all matching node paths of: node_weight multiplied by rarity_boost multiplied by path_decay multiplied by edge_confidence. Each term is defined precisely.
node_weight is the TF-IDF term frequency of that node within the candidate card's trait graph — how central that node is to that card. It is read from the tfidf_vectors table rather than recomputed per query.
rarity_boost is the inverse document frequency component: log(total_cards / cards_containing_node). A node like 'Ben 10 - Alien X' shared by 50 users across 10,000 total cards contributes log(200) = 5.3 to the boost. A node like 'likes music' shared by 8,000 users contributes log(1.25) = 0.22. This single term is what makes the matching feel intelligent rather than generic.
path_decay is 0.85 raised to the power of hop_count. A direct explicit edge (hop 0 from seed) has decay 1.0. A two-hop path has decay 0.72. A three-hop path has decay 0.61. This prevents distant, weakly-connected cards from appearing hotter than directly-connected ones.
edge_confidence is the confidence value on the edge row. Explicit user-input edges have confidence 1.0. System-inferred edges have confidence between 0.3 and 0.7 depending on how strongly the ontology job promoted them. A path through an inferred edge is automatically discounted relative to a path through an explicit edge.
The raw score is then normalized by dividing by the maximum score observed across all candidate cards in this traversal window. This produces a value between 0 and 1. Scores 0 to 0.3 map to cold. Scores 0.3 to 0.6 map to warm. Scores 0.6 to 1.0 map to hot. React Flow reads this normalized score and maps it to node fill color via a linear interpolation: teal at 0.0 interpolating through amber to coral at 1.0.
6.3 score_card SQL Function
The traversal and scoring is wrapped in a single Postgres function named score_card(seed_node_ids uuid[], viewer_card_id uuid). This function executes the recursive CTE, computes per-card scores, normalizes, and returns a table of (card_id uuid, score float4, temperature text). The Next.js Route Handler calls this function with a single Neon serverless driver invocation. One HTTP round-trip from the browser produces the full scored candidate set. The function result is cached in React state for the current traversal session and only re-fetched when seeds change.
6.4 Indexes for Traversal Performance
Three indexes make the recursive CTE fast. A composite btree index on edges(source_id, target_id, edge_type) covers the join from node to neighboring node. A btree index on card_nodes(node_id, card_id) covers the join from node to card. A btree index on card_nodes(card_id, node_id) covers the reverse lookup when building the aggregation. With these indexes, a three-hop traversal over 100,000 cards returns in under 200 milliseconds on Neon's free tier infrastructure.

7. TF-IDF Vectors and Cosine Similarity
7.1 Vector Construction
Each identity card is represented as a sparse float4 array in Postgres. The dimensions of the array correspond to node IDs mapped to integer indices via a stable node_index lookup table (node_id uuid, index integer generated always as identity). Each position in the vector holds the TF-IDF weight for that node relative to that card.
TF (term frequency) for a card-node pair is computed as the count of that node's appearances in the card's trait graph divided by the total number of nodes in the card. A card with 20 nodes where 'Friends' appears twice has TF(Friends) = 2/20 = 0.1. Because most nodes appear exactly once, TF is typically 1/total_nodes, making it a smoothed indicator of card density rather than raw frequency.
IDF (inverse document frequency) is computed globally as the natural log of (total_identity_cards / count of cards containing this node). This is stored in a materialized view refreshed hourly by the background job. The materialized view is named node_idf and has columns (node_id uuid, idf float4, computed_at timestamptz).
The TF-IDF weight for a card-node pair is TF multiplied by IDF. The full sparse vector for a card is assembled by the background job as a float4 array, with zeros at all positions corresponding to nodes the card does not contain. The vector norm (square root of the sum of squares of all weights) is precomputed and stored in the tfidf_vectors.vector_norm column to avoid recomputation at query time.
7.2 Cosine Similarity Query
Given two card vectors A and B, cosine similarity is (A dot B) divided by (|A| times |B|). In SQL, the dot product of two float4 arrays is computed using the Postgres intarray extension's array overlap operations, or more precisely via a set-returning function that unnests both arrays in parallel and sums the products of matching indices. The magnitudes are read directly from the precomputed vector_norm column. The full similarity between two cards is therefore a single SQL expression that reads two float4 arrays and two float4 scalars and returns a float4 in the range -1 to 1 (in practice 0 to 1 for non-negative TF-IDF weights).
This is used in two places. First, in the Finder search bar: when a Finder types a query, the query is parsed into seed nodes, and cosine similarity is computed between a synthetic query vector (built from those seed nodes' IDF weights) and all public card vectors. The top-k results by cosine similarity are returned as the initial candidate set before the recursive CTE traversal refines them. Second, in the recommendation layer: after the hot-cold traversal, the top-10 cards by composite score are re-ranked by cosine similarity to break score ties and surface the most semantically similar cards first.
7.3 K-Means Clustering for Ontology
Every 24 hours, a Postgres background job runs k-means clustering over the tfidf_vectors table. K-means is implemented as an iterative SQL procedure. The procedure initializes k centroids by selecting k random card vectors. Then it iterates: it assigns each card to its nearest centroid by cosine distance, recomputes each centroid as the mean vector of its assigned cards, and checks for convergence by computing the sum of centroid displacement across iterations. Convergence is declared when total displacement falls below a threshold or after a maximum of 50 iterations.
The cluster assignments are written to a card_clusters table (card_id, cluster_id, assigned_at). The centroid vectors are written to a cluster_centroids table (cluster_id, centroid_vector float4[], size integer). Nodes that appear at high TF-IDF weight across many cards within the same cluster are identified as cluster-defining nodes. If two such nodes consistently co-occur at the cluster level but do not yet have an explicit edge connecting them in the edges table, the job creates a soft inferred edge between them with confidence 0.4. This is how Jim Halpert and sarcasm become connected without any user explicitly typing that relationship.

8. Ontology Inference Engine
8.1 Co-Occurrence Mining
The ontology inference job runs as a scheduled Postgres procedure triggered by pg_cron every six hours. It reads the card_nodes table and computes pairwise co-occurrence counts: for every pair of nodes (A, B) that appear together on at least one card, it counts how many distinct cards contain both. These counts are written to the node_cooccurrence table using INSERT ... ON CONFLICT DO UPDATE to accumulate over time without reprocessing historical data.
The promotion threshold is computed as follows. Let P(A and B) = co_count / total_cards_seen. Let P(A) = cards_with_A / total_cards. Let P(B) = cards_with_B / total_cards. The pointwise mutual information (PMI) between A and B is log2(P(A and B) / (P(A) times P(B))). PMI above 2.0 and co_count above 100 triggers promotion of a soft inferred edge from A to B with confidence min(0.3 + (PMI / 10), 0.7). PMI captures directionality: if every Jim Halpert fan also likes sarcasm but not every sarcasm fan likes Jim Halpert, the PMI from Jim to sarcasm is high while the reverse PMI is lower, correctly reflecting the asymmetric relationship.
8.2 Edge Hardening
Soft inferred edges start with confidence 0.4 and are tagged source_of_truth = system_inferred. They are shown to Finders as dashed lines in the graph canvas, visually distinguishing them from solid explicit edges. As more users independently add both nodes to their cards — increasing co_count — the confidence is updated by the job up to a ceiling of 0.7. Crossing 500 co-occurrences with PMI above 3.0 promotes the edge to confidence 0.7 and changes the visual to a semi-solid line. Only reaching 2000 co-occurrences with PMI above 4.0 can promote an edge to confidence 1.0, at which point it is considered ontologically validated and rendered as a solid line indistinguishable from explicit edges.
8.3 Derived Concept Nodes
When a cluster from the k-means job contains more than 200 cards and has a stable centroid across three consecutive daily runs, the job creates a derived concept node. The derived node's display_name is generated from the top-3 highest-IDF nodes in the centroid vector, concatenated as a descriptive label (e.g., 'Dry Humor / Sitcom Fan / Strategic Characters'). The derived node is given type = derived and connected via edges to all top-10 nodes in its centroid. These derived nodes become navigational shortcuts in the graph canvas — clusters that users can zoom into to find a concentrated community of similar cards.

9. Search System
9.1 Full-Text and Fuzzy Search
Search at MVP lives entirely in Postgres. The pg_trgm extension enables trigram-based fuzzy matching on node display names and aliases. The tsvector column on search_terms enables ranked full-text search. A search query runs both in parallel using a UNION of the two result sets, deduplicated by node_id, and ranked by a composite score of tsvector rank plus trigram similarity plus node rarity (IDF from the materialized view).
The Route Handler at GET /api/search?q= receives the query string, sanitizes it (strip special characters, lowercase, trim), and executes this composite query against Neon. Results include node type, slug, display name, and a relevance score. The autocomplete dropdown in the UI is populated from the top-10 results returned within 80 milliseconds — achievable because pg_trgm with a GIN index is extremely fast on name-length strings.
9.2 Semantic Query Expansion
When a Finder submits a full search (not just autocomplete), the query goes through a second expansion step. The matched seed nodes are looked up in the edges table for their top-5 highest-confidence neighbors. Those neighbors are added to the seed set at a discount weight of 0.5. This means searching for 'Jim Halpert' also implicitly seeds the traversal with 'sarcasm,' 'pranks,' and 'Dunder Mifflin' — because those edges exist in the ontology — without the Finder having to type them. The expanded seed set is then passed to the score_card function for the full hot-cold traversal.
9.3 Asynchronous Index Updates
When a node is created or updated, a Server Action writes to the audit_events table with event_type = node_mutated. A Vercel Edge Function scheduled to run every 60 seconds reads unprocessed node_mutated events, rebuilds the search_terms rows for those node_ids, and marks the events as processed. This keeps search consistent with graph state without blocking the mutation path. The 60-second lag is acceptable for a discovery platform where freshness is not real-time critical.

10. Frontend Architecture
10.1 Three UI Surfaces
The frontend is divided into three surfaces that share the same underlying graph state. The Graph Canvas is the primary exploration interface, rendered using @xyflow/react. The Identity Card Editor is the structured form for Hiders. The Finder Rail is the search and traversal control panel that feeds seeds into the canvas.
All three surfaces consume from a single React context that holds the current graph neighborhood: the seed node IDs, the candidate card scores from score_card, the active card detail, and the traversal history. State mutations flow through Server Actions. Read state flows through React Server Components that fetch from Neon at the edge. The result is that the page is fully server-rendered on initial load with zero client-side fetching delay, and subsequent interactions update only the relevant state slices via Server Actions without full-page reloads.
10.2 Graph Canvas
React Flow renders nodes as custom React components. Each node is sized by its score: a hot card gets a larger, brighter node radius. The node color is a linear interpolation over the teal-amber-coral ramp keyed to the normalized 0-1 score. Edge rendering uses React Flow's built-in edge types: explicit ontology edges render as solid curves, inferred edges as dashed curves, and active traversal paths as animated dashed lines with a moving dash offset driven by a CSS keyframe animation.
The canvas supports zoom, pan, node selection, and cluster zoom. When a user zooms out past a threshold, individual card nodes collapse into cluster nodes (derived concept nodes from the ontology) showing only the cluster label and card count. Zooming in expands the cluster into individual cards. This progressive disclosure is implemented by switching the React Flow nodes array between card-level data and cluster-level data based on the current viewport transform scale, with a smooth Motion layout animation on the transition.
10.3 Hot-Cold Feedback Animation
When the Finder moves the traversal — clicking a node, adding a seed, or expanding a thread — the score_card function is called and new scores arrive. Motion's layout animations handle the visual transition. Nodes that increased in score animate their fill color from cool to warm over 400 milliseconds using Motion's animate prop on the SVG fill attribute. Nodes that dropped in score animate the reverse. The temperature label (cold, warm, hot) updates instantly as the new score arrives, while the color transition happens smoothly. This creates the sensation that the graph is reacting to the user's movement — pulling warmer cards forward and pushing colder ones back.
10.4 Thread Pulling
Thread pulling is the core interaction mechanic. A Finder selects an edge in the canvas and drags it outward. This gesture calls a Route Handler that extends the traversal one hop further along that edge type, fetching the next neighborhood. The result is appended to the React Flow nodes and edges arrays, and Motion animates the new nodes appearing from the direction of the dragged edge. The sensation is of physically pulling a thread and watching new cards emerge from the fabric of the graph. Technically this is a standard React Flow edge interaction event that triggers a fetch and a state update, but the animation timing — 200ms delay, 350ms ease-out entrance — makes it feel physical.

11. Friend Hunt and Challenge Links
The friend-hunt mode is a constrained traversal. A Hider creates a challenge by selecting a subset of their card's nodes as clues and calling a Server Action that generates a challenge_links row. Fields: id (uuid), creator_card_id (uuid), clue_node_ids (uuid[]), expires_at (timestamptz, default 72 hours), created_at. The challenge link URL encodes the challenge id. Anyone with the link — logged in or anonymous — can open the challenge.
The challenge interface shows only the selected clue nodes as seeds. The Finder traverses the graph normally, but the hot-cold scores are computed relative to the creator's card as the hidden target. When the Finder reaches a card with a score above 0.85, the system reveals that they have found the correct card and shows a match confirmation. The creator is notified via a Server Action that writes to audit_events. The challenge link expires after 72 hours. This entire mode reuses the score_card function, the graph canvas, and the hot-cold animation system — it is purely a constrained seed set with a hidden target.

12. Deployment — Free for One Year
12.1 Vercel Hobby
The Next.js application deploys to Vercel Hobby at zero cost. Vercel Hobby is a permanent free tier, not a trial. It provides: global edge network deployment across 30+ regions; automatic HTTPS with a vercel.app subdomain; unlimited bandwidth for typical early-stage traffic; Vercel Edge Functions for the scheduled search index update job; build time up to 45 minutes per deployment; and up to 100 deployments per day. Custom domains can be added at any time by pointing a DNS A record to Vercel's IP — the domain itself costs roughly $10 per year from Cloudflare Registrar, which is the only cost in the entire stack.
12.2 Neon Free Tier
Neon provides the Postgres database. The free tier includes: 0.5 GB of storage (sufficient for tens of thousands of identity cards with their node and edge data); no connection limit via the Neon serverless HTTP driver used by Next.js; no inactivity pausing (the critical improvement over Supabase's free tier, which pauses after one week of inactivity); one compute endpoint with 0.25 vCPU and 1 GB RAM; and pg_cron for scheduled jobs. The Neon serverless driver (@neondatabase/serverless) connects over HTTP rather than TCP, which is compatible with Vercel's serverless function execution model.
12.3 Cloudflare R2
Avatar storage uses Cloudflare R2. The free tier includes 10 GB of storage and 1 million Class A operations (uploads) and 10 million Class B operations (reads) per month. Egress from R2 to the internet is free, which is the primary reason to use R2 over S3. The R2 bucket is connected to a Cloudflare custom domain for avatar serving, providing CDN-level performance at zero cost.
12.4 Resend
Email delivery for magic links and notifications uses Resend's free tier: 100 emails per day, 3000 per month. This is sufficient for MVP-scale user acquisition. The Resend SDK is integrated into Better Auth's email provider configuration with four environment variables: RESEND_API_KEY, EMAIL_FROM, EMAIL_FROM_NAME, and the Better Auth base URL.
12.5 Environment Variables
The complete list of environment variables required: DATABASE_URL (Neon connection string with pooling), BETTER_AUTH_SECRET (random 32-byte hex string), BETTER_AUTH_URL (the Vercel deployment URL), GITHUB_CLIENT_ID and GITHUB_CLIENT_SECRET (from GitHub OAuth app), RESEND_API_KEY, EMAIL_FROM (verified sender address), R2_ACCESS_KEY_ID, R2_SECRET_ACCESS_KEY, R2_BUCKET_NAME, and R2_PUBLIC_URL. All are set in the Vercel project dashboard under Environment Variables. None are committed to the repository.

At zero users: $0/month. At 1,000 monthly active users with average 50 nodes per card: approximately 50,000 node rows, 200,000 edge rows, 150 MB of Neon storage — well within free tier. At 10,000 monthly active users: approximately 500,000 node rows, 2 million edge rows, 1.2 GB of Neon storage — requires Neon Launch plan at $19/month, Vercel Pro at $20/month if serverless function execution exceeds the Hobby limit. The first year at typical indie-project growth stays at $0 or at most $10 for a custom domain. The architecture is stateless on the frontend and indexed in the database, so scaling from $0 to paid tiers requires no architectural changes — only configuration upgrades.

13. Repository Structure
The repository is a single Next.js monorepo. The directory structure is: app/ containing the App Router pages and layouts; app/api/ containing Route Handlers for score_card, search, and challenge resolution; app/actions/ containing all Server Actions for mutations; components/ containing React components split into graph/ (canvas, nodes, edges), identity/ (card editor form), and finder/ (search rail, thread controls); lib/ containing auth.ts (Better Auth config), db.ts (Drizzle + Neon connection), schema.ts (all Drizzle table definitions), and vectors.ts (TF-IDF and cosine similarity utilities); jobs/ containing the TypeScript implementations of the ontology inference job and the k-means clustering job, executed via pg_cron calling Postgres procedures; and drizzle/ containing migration files generated by drizzle-kit.
Drizzle is the single source of truth for the schema. Running npx drizzle-kit generate produces SQL migration files. Running npx drizzle-kit migrate applies them to Neon. The schema.ts file defines every table using Drizzle's TypeScript schema DSL, providing full type safety from the database through to the React components without any manual type declarations.
